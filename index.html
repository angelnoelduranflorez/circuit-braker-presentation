Resiliencia en Software

"Capacidad de un sistema de software para anticipar, resistir, recuperarse y adaptarse a condiciones adversas, ataques o fallos"

En la práctica significa:

  • El sistema sigue funcionando ante fallos parciales
  • Se degrada elegantemente en vez de caer completamente
  • Se recupera automáticamente cuando la condición adversa desaparece
✅ Estado Normal
↓ Fallo ocurre
⚠️ Degradado
↓ Auto-recuperación
✅ Recuperado

Pruebas de Resiliencia

🐒 Chaos Engineering

Disciplina de experimentar en un sistema para construir confianza en su capacidad de soportar condiciones turbulentas en producción.

Herramientas populares:

  • Chaos Monkey (Netflix)
  • Gremlin
  • LitmusChaos (Kubernetes)
  • AWS Fault Injection Simulator

Tipos de pruebas:

  • 💀 Inyección de fallos de red
  • 🐌 Introducción de latencia
  • 💾 Saturación de recursos
  • 🔌 Caída de instancias
  • 🌍 Pérdida de zona de disponibilidad

Aplicación en Entornos Distribuidos

Complejidad de los sistemas distribuidos

  • Múltiples servicios comunicándose por red
  • Cada llamada remota puede fallar
  • Latencia variable e impredecible
  • Fallos parciales son la norma, no la excepción

Fallos comunes en microservicios

  • 🌐 Timeouts de red
  • 💾 Servicios sobrecargados
  • 🔗 Dependencias en cascada
  • 📊 Agotamiento de thread pools

Mecanismos de resiliencia:

Circuit Breaker, Retry, Timeout, Bulkhead, Rate Limiter, Fallback

¿Qué es Circuit Breaker?

Patrón de arquitectura de microservicios que previene que un cliente invoque de forma continua a un servicio que está fallando o que tiene un problema de performance.

⚡ Analogía Eléctrica

Como un interruptor térmico en tu casa: cuando detecta una sobrecarga, corta el circuito para proteger toda la instalación.

🖥️ Cliente
⚡ Circuit Breaker
🌐 Servicio

¿Por qué es importante?

1

Evita Fallas en Cascada

Un servicio lento no arrastra al resto del sistema. El circuit breaker corta la cadena de propagación.

2

Libera Recursos

En vez de mantener threads bloqueados esperando respuesta, se rechazan inmediatamente.

3

Permite Recuperación

El servicio fallido tiene tiempo para recuperarse sin recibir tráfico adicional.

¿Por qué es importante?

4

Mejora la Experiencia del Usuario

Respuestas rápidas (aunque sean fallback) son mejores que timeouts de 30 segundos.

5

Observabilidad

Métricas claras sobre la salud de las dependencias. Alertas proactivas.

6

Resiliencia del Ecosistema

Protege no solo tu servicio, sino todo el ecosistema de microservicios interconectados.

¿Cómo funciona un Circuit Breaker?

✅

CLOSED

Funcionamiento normal. Las solicitudes pasan al servicio.

Umbral de errores superado
🚫

OPEN

Circuito abierto. Solicitudes rechazadas inmediatamente.

Timeout expira
⚠️

HALF-OPEN

Permite solicitudes limitadas para probar recuperación.

¿Cómo funciona un Circuit Breaker?

1

CLOSED → Monitorea

Cuenta las solicitudes exitosas y fallidas. Mientras el ratio de error esté bajo el umbral, todo pasa normal.

2

CLOSED → OPEN

Cuando el porcentaje de fallos supera el umbral configurado, el circuito se abre. Todas las solicitudes son rechazadas de inmediato (fail-fast).

3

OPEN → HALF-OPEN

Después de un timeout configurable, el circuito pasa a half-open. Permite un número limitado de solicitudes de prueba.

4

HALF-OPEN → CLOSED / OPEN

Si las pruebas son exitosas → vuelve a CLOSED. Si fallan → vuelve a OPEN y reinicia el timeout.

Enfoques de Funcionamiento

📊

Count-based (por conteo)

Se basa en un número fijo de llamadas. El circuit breaker evalúa las últimas N llamadas. Si el porcentaje de fallos supera el umbral, se abre.

failureRateThreshold: 50%
slidingWindowType: COUNT_BASED
slidingWindowSize: 10
⏱️

Time-based (por tiempo)

Se basa en una ventana de tiempo. Evalúa todas las llamadas realizadas en los últimos N segundos. Si el % de errores supera el umbral, se abre.

failureRateThreshold: 50%
slidingWindowType: TIME_BASED
slidingWindowSize: 10 (seconds)

🔄

Limitaciones del Circuit Breaker

⚙️

Complejidad adicional

Añade una capa más al código que hay que mantener y entender

🎛️

Tuning delicado

Configurar umbrales incorrectos puede causar más problemas que los que resuelve

🚨

Falsos positivos

Puede abrir el circuito por picos transitorios que no son un problema real

🔍

No resuelve la causa raíz

Solo mitiga el impacto; el fallo subyacente requiere otra solución

🧪

Testing complejo

Probar transiciones de estado y escenarios de borde requiere más esfuerzo

Ejemplo Práctico: Python

import pybreaker
import requests

# Configurar el Circuit Breaker
breaker = pybreaker.CircuitBreaker(
    fail_max=5,           # Máximo de fallos antes de abrir
    reset_timeout=30,      # Segundos para pasar a half-open
    exclude=[ValueError]    # Excepciones que NO cuentan como fallo
)

@breaker
def call_payment_service(order_id):
    """Llama al servicio de pagos protegido por CB"""
    response = requests.post(
        f"https://payments.api/process/{order_id}",
        timeout=5
    )
    response.raise_for_status()
    return response.json()

# Uso con fallback
try:
    result = call_payment_service("ORD-12345")
except pybreaker.CircuitBreakerError:
    # Circuito abierto → fallback
    result = {"status": "queued", "msg": "Procesando más tarde"}

Ejemplo Práctico: Java

// Configuración del CircuitBreaker con Resilience4j
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)              // 50% de fallos abre el circuito
    .waitDurationInOpenState(Duration.ofSeconds(30))
    .slidingWindowSize(10)                  // Últimas 10 llamadas
    .permittedNumberOfCallsInHalfOpenState(3)
    .build();

CircuitBreaker cb = CircuitBreaker.of("paymentService", config);

// Decorar la función con Circuit Breaker
Supplier<PaymentResponse> decorated = CircuitBreaker
    .decorateSupplier(cb, () -> paymentService.process(order));

// Ejecutar con fallback
PaymentResponse result = Try.ofSupplier(decorated)
    .recover(CallNotPermittedException.class,
        e -> PaymentResponse.queued("Servicio no disponible"))
    .recover(Exception.class,
        e -> PaymentResponse.error(e.getMessage()))
    .get();
